# 🏡 Lumière - AI 智能家装规划师系统说明文档

本项目（AI Home Renovation Planner Agent）是一个整合了多模态大模型和多智能体技术的现代家装规划与设计生成平台。

---

### 1. 前端架构及特点
**架构技术栈**：采用纯粹的前端分离架构，基于 **Next.js 13.4.x** + **React 18** 构建，样式管理使用 **Tailwind CSS**，并整合了 Framer Motion 动画库。

**核心特点**：
- **双模工作流设计**：在系统的主要工作区 `/dream` 当中，提供了两种交互模式（`generate` 快速生成模式 和 `chat` 自由交互模式），迎合不同需求深度的用户。
- **现代化组件解耦**：细化了诸多可复用组件，例如专门用于追踪 Agent 运行状态的 `AgentStatus` 组件，管理上下文对话的 `ChatHistoryPanel`，以及高交互拖拽上传的 `UploadDropZone`。
- **用户引导友好**：封装了 `QuickPrompts` (快捷词) 和 `QuickScenes` (预设风格场景) 工具，降低冷启动门槛。

### 2. 后端架构及特点
**架构技术栈**：基于 **Python** + **FastAPI** 框架，提供纯净的 RESTful API + SSE流式通讯 接口服务。

**核心特点**：
- **完全 API 化与解耦**：所有的核心 AI 推理逻辑剥离为底层服务（Server.py），无缝处理前端的文字指令、多模态图片上传及流事件响应。不再有复杂的耦合，极易扩展到小程序或移动端等。
- **持久化及资产隔离**：通过本地 SQLite (`.adk/planner.db`) 进行用户会话信息和对话流水的留存追踪；并通过专门的 Artifact 管理策略存储大模型所产出的图像资源 (`.adk/artifacts/`)。
- **异步处理保障体验**：针对生成渲染图这种高耗时任务，后端采用 LangGraph 的主图 `graph` 与独立渲染子图 `render_graph` 分离执行，避免主请求阻塞。

### 3. 后端集成的多 Agent 机制
系统彻底摒弃了常规“单一提示词”的做法，采用了 **LangGraph** 实现高级的 **Coordinator/Dispatcher + 顺序流水线流水线 (Sequential Pipeline)** 设计模式：
- **共享状态 (RenovationState)**：不同智能体间通过一个中心化的字典交互，内含消息历史、房屋分析、预算约束、当前需求风格等状态上下文。
- **Router Node (协调者/分发器)**：作为前置关口，自动分类用户的输入。若为普通解答指派给 `InfoAgent`，若附带新图片或设计需求则路由进入 `PlanningPipeline` (规划流水线)，若需修改图片则进入 `RenderingEditor`。
- **流水线智能协同（主对话图）**：
   - `VisualAssessor`（视觉评估师）：调用多模态模型负责看图，提取房屋现有户型、软装现状及找出的待变点。
   - `DesignPlanner`（设计规划师）：基于上一步输出，撰写整体设计方案和材料预算等。
- **独立渲染子图（后台生成）**：
   - `ProjectCoordinator`（项目协调员）：位于 `render_graph` 子图中，通过渲染任务接口触发，专门负责调用生图/改图工具并返回渲染结果。

### 4. 平台模型调用方案与解耦设计
本项目的模型深度耦合在 `llm_provider.py` 中，通过轻量适配层以极致解耦的方式实现。
*(注：项目 `requirements.txt` 列表中虽然预留有 `litellm` 的库依赖，但这可能是前期开发或为了配合本地代理预留的。在当前的最终架构下，系统已经**无需通过 LiteLLM 后台作为中转网关转发**，而是采用直接面向提供商的原生直连机制。)*

**核心调用方法与工具**：
- **LangChain ChatOpenAI 原生直连**：系统直接利用了 `langchain_openai` 库的 `ChatOpenAI` 类。由于现今如阿里云（Qwen/通义千问）等各大厂商皆已全面支持 OpenAI API 报文协议，系统通过直接覆写 `openai_api_base`，以极其标准的 OpenAI 调用方法平滑接入各家国产大模型生态。
- **环境变量动态配置**：系统不绑定任何特定的代码层域名，而是完全通过读取 `.env` 来指派模型走向（彻底实现了业务的解耦和模型的即插即用）。

**目前运行阶段实际调用的主要大模型与平台（基于 .env 配置）**：
1. **纯文本逻辑推理模型**：
   - **使用模型**：`qwen3-max` （通义千问3代旗舰版）  
   - **所处平台**：阿里云百炼 / DashScope 平台（调用接口兼容基址：`https://dashscope.aliyuncs.com/compatible-mode/v1`）
   - **核心职责**：在 `Router` 中作为网关分发意图，并且作为核心的业务大脑 `Design Planner` 来完成所有的文本规划与任务推演。
2. **多模态视觉图文模型**：
   - **使用模型**：`qwen3-vl-plus` （通义千问3代多模态视觉增强版）  
   - **所处平台**：阿里云百炼 / DashScope 平台
   - **核心职责**：担任视觉评估师 (`Visual Assessor`)。使用 Base64 图片协议提取原实景图中诸如门窗结构、拍摄视角和现软硬装等复杂视觉要素。
3. **图像生成与渲染大模型**：
   - **使用模型**：`wan2.7-image`
   - **所处平台**：阿里云百炼 / DashScope 图像生成平台（使用基址：`https://dashscope.aliyuncs.com/api/v1`）
   - **核心职责**：接管最后一步，负责执行由逻辑大模型重组好的强化结构约束提示词，真正生成出照片级照片质量的室内改造图。

### 5. 可用能力集与网络搜索实现
智能体在运行中可以灵活挂载不同的工具（存储在 `tools.py`）。
- **网络搜索实现**：系统废弃了原始架构的 Google CSE 谷歌搜索，转而开发了 **`baidu_search_tool`**，其底层 (`baidu_search.py`) 采用**百度千帆大模型平台的 AI Web Search** 作为搜索引擎底座。不仅合规且能获取基于国内语境及审美的实效知识。
- **其他关键 Tool (工具)**：
  - `estimate_renovation_cost_tool` (自动预估工价物料与装修成本)
  - `calculate_timeline_tool` (根据工作范围运算工序推进排期表)
  - `generate_renovation_rendering_tool` (结合当前图与灵感调用绘图服务渲染设计结果图)
  - `edit_renovation_rendering_tool` (对渲染出的成品进行打回/重画及局部调整)

### 6. 本项目的功能及应用场景
**核心功能**：
- 智能扫描照片空间提取结构、测算尺寸及缺陷；
- 在设定资金限额（预算控制）内自动规划高性价比的硬装修与软装升级路线；
- 产出高保真级目标渲染效果图；
- 统筹规划，输出含所需承包商名录、工作时间表和项目行动清单。

**应用场景**：
- **普通房主**：作为低成本前期的个人房屋翻新试错和规划平台、“改造避坑指南”。
- **家装设计师**：可快速向潜在业主推演不同风格下“当前空间”变换后的效果抓手。
- **家居内容创作者**：实现室内设计风格自动迁移配图以及预算预估输出。

### 7. 本项目的特色或创新点
- **Structure-locked 的高阶生图提示机制**：在出图核心逻辑（`_build_structure_locked_prompt`）内写死了极高强度的业务结构约束。**确保AI渲染图具有“落地指导意义”**，而不是让大模型天马行空地新增窗户、打破承重墙或是变更关键机位，在行业痛点上做了针对性改进（仅允许非构架修改，如软装、材质、灯光色调）。
- **国内场景环境强自适应**：整个项目深入进行了本地化适配改造（例如 Prompt 内部强制要求智能体使用“平米”、“人民币”，在生成建议时优先考虑国内家具采购逻辑，乃至采用百度搜索引擎补全资料），显著提高了输出文本对中国用户的友好度。
- **白盒化的过程感知界面**：不再是黑盒子的等待模式。基于智能体拆解，前端界面能直接显示目前是哪个专业角色的 Agent 正在为您分析，这极大提高了系统的专业感和可信度。

### 8. 其他重要且值得关注的地方
- **细粒度词汇及品牌平替映射**：为了适配底层英文原模的生成习惯偏差，系统（如 `server.py` 内）预构建了庞大的特定术语正则替换表 (`HEADING_REPLACEMENTS`/`BRAND_REPLACEMENTS`)。把例如老外习惯的国外建材品牌，在最终给用户呈现前替换为“国内通识性的高品质乳胶漆、现代家具品牌等描述”。
- **无障碍降级与兼容**：具备灵活的错误处理（如 `UPSTREAM_503_MAX_RETRIES` 的网络指数重试避让逻辑），并在图像生成环节预留 `local_mock` 开发配置等，提高了工程鲁棒性和容错能力。
